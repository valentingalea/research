[New Language Features]

uniform initialisation (VS2013+/GCC4.4+)

C++03 had multiple (and inconsistent) methods to declare/initialise data:

  • <nothing> - default initialisation 
int i;             // integer with undefined value
Widget w;          // calls default constructor if any
  • () or (arg, ...) - default or direct initialisation 
int i = int();     // zero-init integer
int f(5);          // an integer with value 5
Widget();          // calls default contructor, temp object
int f();           // Error: "most vexing parse"
                   // this is a function declaration: int (*)(void)
;
Widget w(param);   // OK: copy/convert construction if param is variable
Widget w(Param()); // Error: if Param is type - function declaration again
  • { item1, ... } - array or aggregate initialisation
    • aggregate is class without constructors, private/protected, bases, virtual

uniform initialisation (continued)

C++11 introduces the {} that can be used in all these cases:

int{};
int i{};             // OK: not function declaration
Widget w{};          // OK: again
Widget w{other};     // OK: copy or conversion construction
Widget w = other;    // OK: still construction, not assignment
Widget w = {other};  // OK: the same
int arr[]                   = { 1, 2, 3, 4, 5 };
int matrix[2][2]            = { {5, 6}, {7, 8} };
std::vector<int> v          = { 1, 2, 3, 4, 5 };
std::map<int,std::string> m = { {0, "zero"}, {1, "one"}, {2, "two"} };
struct Widget {
    int i;
    double d;
}
;
Widget factory() {
    return {1, 1.0};
}

uniform initialisation (continued)

Note
 Doesn’t do automatic narrowing conversion! 
int i = 1.0;        // OK: implicitly converts from float to int
int i = { 1.0 };    // Error: won't compile
float f = { 3 };    // OK: not narrowing just another way of saying 3.0
Tip
 Leverage the initializer list: 
struct Widget {
    Widget(std::initializer_list<int> list) {
        for (const int *i = list.begin(); i != list.end(); ++i) { // no [] operator
            *i; // do some work...
        }
    }
}
;
Widget w { 0, 1, 2, 3 }; // technically should be: { {0, 1, 2, 3} }
                         // NOTE: these type of ctors will take precedence over others

auto (VS2010+/GCC4.4+)

The type of the variable that is being declared will be automatically deduced from its initializer.

 int j = 0;         // Variable j is explicitly type int.
auto k = 0;         // Variable k is implicitly type int because 0 is an integer.
auto d = 3.14;      // Variable d is double
auto f = 3.14f;     // Variable f is float
map<int, list<string>>::iterator i = m.begin();
                            auto i = m.begin();
int data;
auto value = data;        // type is: int
auto &ref = data;         // type is: int &
auto ptr = &data;         // type is: int *
auto *ptr = &data;        // type is: int *
auto i = { 42 };          // type is std::initializer_list<int> !!!

auto (continued)

The rules of this deduction are the same of T from template declaration.

auto a = expr;    /* is actually: */   template<class T> void temp(T a) {}
                                       temp<???>(expr); // compile and extract type of T

This means that it doesn’t deduct(it drops) references and const !!!

float& f1();
auto a1 = f1();         // a1 is value float, not reference!
;
const int f2();
auto a2 = f2();         // a2 is not const!
;
const Widget& f3();
auto a3 = f3();         // a3 is not reference, not const, it will be a copy!!!
auto &a1 = f1();        // fixed!
const auto a2 = f2();   // fixed!
const auto &a3 = f3();  // fixed!

decltype (VS2010+/GCC4.3+)

Inspects the declared type of an entity or queries the type and value category of an expression.

int i;
decltype(i) x;           // type is int
struct A { double x; };
const A* a = new A{};
;
decltype(a) x1;         // type is const A *
decltype(a->x) x2;      // type is double
const int func();
decltype(func()) x1;    // type is const int
decltype(func) x2;      // function declaration
std::vector<int> v = { 0, 1, 2, 3 };
decltype(v)::iterator i = v.begin();

decltype (continued)

decltype will get different results from auto in similar situations:

const int& x;
;
auto a1 = x;           // a1 is int
decltype(x) a2;        // a2 is const int &

C++14 "fixes" this:

decltype(auto) a2 = x; // a2 is const int &

range-based for loop (VS2013+/GCC4.6+)

More readable for loop over a range.

int array[] = {0, 1, 2, 3, 4, 5};
for(int n: array)
    std::cout << n; // access by value (like regular for usage)
std::vector<int> vector = {0, 1, 2, 3, 4, 5};
for(const int &i : vector)
    std::cout << i; // access by reference
for(int n: {0, 1, 2, 3, 4, 5})
    std::cout << n;
std::vector<Big_Object> big_list = {...};
for(auto i: big_list)
    std::cout << i; // DANGER - will value copy every iteration!!!

range-based for loop - roll your own

Works on anything that has .begin() and .end() or the free functions equivalent:

struct MyObject {
    // ...
};
struct MyList {
    typedef MyObject* Iterator;
    Iterator begin();
    Iterator end();
    // ...
};
MyList list;
for (auto &i : list) {
    i; // type is MyObject &
}

constexpr (VS2015+/GCC4.6+)

Declares that it is possible to evaluate the value of the function or variable at compile time.

Excellent replacement for macros: they are full C++ entities with scope and are debug-able if needed.

/* BAD */
#define MAGIC 0xFF04
/* GOOD */
constexpr int MAGIC = 0xFF04;
/* BAD */
#define FUNC(a, b) ((a) + (b) - 123)
/* GOOD */
constexpr int FUNC(int a, int b)
{
        return a + b - 123;
}
// NOTE: can be template

constexpr (continued)

constexpr functions in C++11 are limited in what they can use/do:

  • only one return statement (but supports recursion)
  • no if/while/for/switch
  • no local variables.

C++14 relaxes these requirements a lot - but sadly VS 2015 doesn’t support it

Tip
 Additional info:
  • constexpr means implicitly const and inline
  • a constexpr function with non-literal, non-constexpr arguments will turn into a normal runtime function

strong enum (VS2012+/GCC4.4+)

C++03 enum's have big problems:

  • implicitly convert to an integer, causing errors when that is not desired
  • export their enumerators to the surrounding scope, causing name clashes
  • the underlying type of an enum cannot be specified 
enum Alert { green, yellow, red };  // what's the type?
Alert r = Alert::red;               // Error, but why?
int b = red;                        // OK, but where is it coming from and why convert to int?

The new C++11 enum alternative:

enum class NewAlert { green, yellow, red };  // C++11 scoped, strong enum
enum class Direction : char { left, right }; // custom underlying type, default is int
NewAlert r = NewAlert::red;                  // OK now
int q = NewAlert::red                        // Error: no conversion to int any more
Direction d = static_cast<Direction>(1);     // OK with explicit cast

noexcept (VS2015+/GCC4.6)

In C++03 functions declarations could specify their exceptions via throw() statements, but it never caught on and it had problems.

void func() throw(std::exception, int);  // the list of potential exceptions that trigger inside func

Since C++11 there is a way simpler method: noexcept

void func() noexcept;                    // this function will never throw exceptions
void func() noexcept(true);              // same thing
void func() noexcept(<expression>);      // bool constexpr expression for cases when func has exceptions
bool has_errors = noexcept(func());      // operator version for checking

Why care?

  • A function declared with noexcept enables compilers to generate more efficient, optimised code!
  • Some std containers like std::vector won’t move elements unless the move constructor is noexcept!
  • If a noexcept function does throw an exception, std::terminate is called immediately

syntactic sugar

  • using - type alias equivalent to typedef but better 
using my_int = int;                    typedef int my_int;
using func = void (*) (int,int);       typedef void (*func)(int, int);
template<class T> using pointer = T*;  // no equivalent with typedef
pointer<int> p;                        // alias for int *
  • string literals 
      "normal ASCII string"
 L"wide char string"
u8"string encoded in UTF-8"
 u"string encoded in UTF-16"
 U"string encoded in UTF-32"
 R"delimiter( string with anything unescaped like \ / " )delimiter" // delimiter can be any 16 character string, including empty
    • the R one (raw string) can be combined with the others (to specify Unicode etc)
    • they all (if same type) support multi-line
    • Unicode versions are available VS2015+/GCC4.4
    • raw string available VS2013+/GCC4.5

syntactic sugar (continued)

  • binary base numbers (VS2015+/GCC4.9+) 
int i = 0b001110101;
int j = 0B1111;
  • ' (single quote) digit separator (VS2015+/GCC4.9+) 
int i = 1'000'000;
float f = 3'154.632'524f;
int h = 0xFF'FF'FF'FF;

nullptr (VS2010+/GCC4.6+)

Defining null pointers in C++03 was hacky and problematic:

void func(int);
void func(int *);
;
func(0);               // which overload to call?
func(NULL);            // same problem because of #define NULL 0

C++11 introduces the nullptr literal which improves pointer checks:

func(nullptr);         // will always call the int* version
;
int *ptr = nullptr;
if (ptr == nullptr) {
}
Tip
 nullptr is of type std::nullptr_t (which is implementation defined)

alignas, alignof (VS2012+/GCC4.8+)

New keywords to control memory alignment for declarations (before it was compiler specific with #pragma etc)

/* every object of type sse_t will be aligned to 16-byte boundary */
struct alignas(16) sse_t
{
  float sse_data[4];
};
/* the array "cacheline" will be aligned to 128-byte boundary */
alignas(128) char cacheline[128];
/* get the alignment of something */
alignof(char)      // returns 1
alignof(int *)     // returns 8 on x64 systems
alignof(sse_t)     // returns 16
/* equivalent statements */
alignas(float)      alignas(alignof(float))

static_assert (VS2010+/GCC4.3+)

Compile time error messages: static_assert (constexpr-expression, error-message);

template<class T>
struct Widget {
    static_assert(sizeof(int) <= sizeof(T), "Type not big enough!");
        // ...
}
;
Widget<char> w;              // compile error!
static_assert(sizeof(unsigned int) * CHAR_BIT == 32, "Unsupported architecture!");
template <class T>
void func(T value) {
    static_assert(std::is_base_of<MyBase, T>::value, "Type must be derived from MyBase");
    // ...
}
;
func(42);                    // compile error: int not derived from MyBase

[Object Construction Improvements]

=default (VS2013+/GCC4.4+)

Better explicit control over the treatment of default constructors and assignment operators.

Leaving it to the compiler to implement the default behaviour is simpler, less error-prone, and often leads to better object code.

struct Widget
{
    Widget() = default; // the default constructor is explicitly stated, but it's not defined
    Widget(int value);
};
Widget(42);             // OK: legal in all C++ versions
Widget();               // Error in C++03 (due to lack of explicit default constructor)
                        // but OK in C++11
Note
 Works on: Default constructor, Copy constructor, Move constructor, Copy assignment operator, Move assignment operator and Destructor

=delete (VS2013+/GCC4.4+)

Used to prohibit calling any function, which can be used to disallow construction, copying or undesired conversions.

struct NonCopyable {
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};
NonCopyable nc{};       // OK
NonCopyable nc = other; // Error: won't compile
struct OnlyDouble {
    void convert(double d);
    template<class T> void convert(T) = delete;
};
OnlyDouble d;
d.convert(3.14);        // OK
d.convert(3);           // Error: won't compile

delegating constructors (VS2013+/GCC4.7+)

In C++03, if you want two constructors to do the same thing, repeat yourself or call "an init() function."

Now constructors can call each other:

class Widget
{
    int number;
public:
    Widget(int new_number)
            : number {new_number}
        {
        }
    Widget()
            : Widget {42}
        {
        }
};

inheriting constructors (VS2015+/GCC4.8)

Non-virtual methods are not automatically available in derived classes.

They have to be brought into scope with using:: and in C++03 constructors couldn’t be.

struct Base {
    Base(int);
    void method(double);
};
struct Derived : public Base {
    using Base::method;      // lift Base's method into Derived's scope -- works in C++03
    using Base::Base;        // lift all Base constructors into Derived's scope -- new in C++11
};
Derived d {42};
d.method (3.14);

override and final (VS2012+/GCC4.7)

New mechanisms for better enforcing and checking polymorphism rules:

struct Base {
    virtual void method_a(char);
    virtual void method_b() const;
    virtual void method_c() final;
            void method_d();
};
struct Derived : Base {
    void method_a(char) override; // OK: overrides B::method_a()
    void method_b() override;     // error: wrong signature - not const
    void method_c();              // error: attempt to override final method
    void method_d() override;     // error: B::method_d() is not virtual
};
  • both final and override are contextual keywords - they can be used in normal declarations (but it’s discouraged)
  • final can also be used for class declaration to disable inheritance

member initialisation (VS2013+/GCC4.6+)

Non-static data members may be initialized at declaration site now:

struct Widget {
    int x = 7;
    int y {8};
    int &ref = x;
};
Widget s;
assert(s.x == 7);
assert(s.y == 8);
struct Widget {
    int x {7};
    Widget() : x {42} {}   // override value from member initialisation
};

VS2010+/GCC4.5+

alternative function declaration (VS2010+/GCC4.4+)

auto SomeFunction(int x, int y) -> int
{
    return x + y;
}

auto here means something different than in type deduction.

C++14 allows the return type to be automatically deduced, no more need for ->

Especially needed in template declarations where the return type cannot be easily determined due to how the C++ parser works:

/* ILLEGAL */
template<class L, class R>
Ret adding_func(const L &lhs, const R &rhs) { return lhs + rhs; } // what type is Ret???
/* OK */
template<class L, class R>
auto adding_func(const L &lhs, const R &rhs) -> decltype(lhs + rhs) { return lhs + rhs; }

lambda - motivation

C++ has some very useful generic functions like std::sort and std::transform but they tend to be cumbersome to use:

struct comp_functor {
    void operator() (int a, int b) {
        return a < b;
    }
}
;
void work(std::vector<int> &v) {
    com_functor f;
    std::sort(v.begin(), v.end(), f);  // good but what if f is used only here
                                       // also a global function works here but same problem
}
void work(std::vector<int> &v) {
    struct comp_functor {
        void operator() (int a, int b) {
             return a < b;
        }
    } f;
    std::sort(v.begin(), v.end(), f);  // Error: not permitted in C++03!!!
}

lambda - motivation (continued)

Delegates are a very useful construct:

struct Weapon : public Item {
    // ...
    void (*OnFire)(Bullet &projectile);
    void (*OnReload)(Clip &ammo);
    void (*OnBulletImpact)(MuzzleEffect &vfx, Vec3 origin, Vec3 normal);
}
;
Weapon weap;
weap.OnFire = // What to put here???
  • global function
  • functor object
  • std::function — a generalisation of functors
  • turn the events into virtual methods and create interface

lambda - example

In C++11 in all the previous scenarios we can use lambda’s:

void work(std::vector<int> &v) {
    std::sort(v.begin(), v.end(), [](int a, int b) {
        return a < b;
    });
}
Weapon weap;
weap.OnFire = [](Bullet &projectile) {
    projectile.explode();
    // ...
};
Player pl;
weap.OnFire = [&pl](Bullet &projectile) {
    pl.fired_gun = true;
    // ...
};

lambda - definition

Anonymous in-place function (or alternatively an object called closure).

/* lambda introducer     param    modifiable   exception  return type */
/* and capture clause    list     spec         spec                   */
            [y]         (int x)   mutable      noexcept   -> int
            {
                int n = x + y;
                return n;
            }
  • [] capture nothing
  • [=] all by value
  • [&] all by reference
  • [v, &r] v by value, r by ref
  • [=, &r] r by ref, rest by value
  • [&, v] v by value, rest by ref
  • [this] capture the instance. provide access to methods and members

lambda - definition (continued)

/* lambda introducer     param    modifiable   exception  return type */
/* and capture clause    list     spec         spec                   */
            [y]         (int x)   mutable      noexcept   -> int
            {
                int n = x + y;
                return n;
            }
  • mutable
    • allows the body to modify the parameters captured by copy, and to call their non-const member functions
  • noexcept(...)
    • conditions under which exceptions may be thrown or that it doesn’t throw
  • -> ...
    • return type. If not present it’s implied by the function return statements (can be void as well)

lambda - what’s the type

  • raw function pointer — if it doesn’t capture 
void (*func)(int, float) = [](int arg1, float arg2){};
  • hidden, unique "ClosureType" that can be only captured with auto 
auto func = [](int arg1, float arg2){};
  • std::function with the signature of the lambda argument types. Useful to pass around but heavyweight (NOTE: uses type erasure which does heap alloc!) 
std::function<int, float> func = [](int arg1, float arg2){};

lambda - C++14 additions

  • Generic arguments: (this effectively turns it into a template function) 
[] (auto first, auto second) {
    return first + second;
};
  • Capture expressions: 
[&ref = x, new_val = val + 1] () {
    // ...
};
Tip
 This especially allows capture by move, which wasn’t possible before 
[x = std::move(x)] () {
    // ...
};

lambda - under the hood

/* simple lambda declaration */
[](int x) -> int {
    return x + 1;
};
/* what the compiler secretly does */
struct ClosureType {
    int operator() (int x) {
        return x + 1;
    }
};

lambda - under the hood (continued)

int y = 42;
int r = 52;
// etc...
/* closure declaration */
[y, &r](int x) -> int {
    return x + y + r;
};
/* what the compiler secretly does */
struct ClosureType {
    int y_;
    int &r_;
    ClosureType(int y, int &r) : y_(y), r_(r) {}
    int operator() (int x) {
        return x + y_ + r_;
    }
};

lambda - under the hood (continued)

/* generic lambda declaration */
[](auto x, auto y) {
    return x + y;
};
/* what the compiler secretly does */
struct ClosureType {
    template<typename X, typename Y>
    auto operator() (X x, Y y) -> decltype(x + y) {
        return x + y;
    }
};

lambda - gotchas

  • Beware of dangling references! 
{
        Widget w = ...
        callback = [&w]() { w.function(); }
}
callback(); // Disaster!
  • Beware of implicit capture of this 
struct Widget {
        int member;
        auto factory() {
                return [=]() { return member; }; // actually: this->member
        }
};
  • Globals aren’t captured - they are used like in normal functions

[Move Semantics]

VS2010+/GCC4.3+

move semantics - lvalues/rvalues

Every C++ statement is either an lvalue or an rvalue.

  • lvalues
    • generally sit on the left of =
    • are always named, persist
    • you can take their address
  • rvalues
    • generally sit on right of =
    • are temporary i.e. they don’t persist beyond the expression that uses them

In C++03 you cannot do much with rvalues, normally they go away after their use (get destructed)

int& ref = 1 + 2;           // Error: not permitted, getting address of temporary
const int& ref = 1 + 2;     // Ok: because it's constant, extends lifetime
                            // but extremely limited use

move semantics - lvalue/rvalue - examples

/*lvalue*/ /*rvalue*/
int x      = 3 + 4;
*ptr;                   // lvalue
&x;                     // rvalue: the address of x
"hello";                // rvalue: literal
int& func();
int func2();
func();                 // lvalue, the call will return a meaningful address
func2();                // rvalue - an 'invisible' temporary is returned
std::string("hello");   // rvalue
new int;                // rvalue
const widget y {};      // lvalue
int i;                  // lvalue
((i < 3) ? i : x) = 7;  // correct statement: the conditional operator returns lvalue

move semantics - rvalue references

C++11 introduces rvalue references: <identifier> &&

int&& ref = 1 + 2;                         // OK now, grabbing temporary
ref = 42;                                  // still alive and kicking!

To leverage this, move constructors and move assignment operator are introduced:

struct Widget {
    Widget(Widget&& other);                // move constructor
    Widget& operator=(Widget&& other);     // move assignment operator
};
Note
 In the context of template deduction && means something different: a "universal reference" used to implement perfect forwarding: that is ability to accept both lvalue and rvalue arguments. 
template<class T> void func(T&& arg);      // not a rvalue reference, but could be!

move semantics - example

struct MemoryPage {
    void * address;
    size_t size;

    // appropriate constructor & destructor
    MemoryPage() {...}
    ~MemoryPage() {...}

    // copy constructor and assignment operator
    MemoryPage(const MemoryPage& other) {
        memcpy(address, other.address, other.size);
        size = other.size;
    }
    MemoryPage& operator=(const MemoryPage&) {
        // same expensive memcpy...
    }

move semantics - example (continued)

    // move constructor and move assignment operator
    MemoryPage(MemoryPage&& other) noexcept {
        // trivial swap of data
        address = other.address;
        size = other.size;
        // reset other, making it valid but empty
        other.address = nullptr;
        other.size = 0;
    }
    MemoryPage& operator=(MemoryPage&& other) noexcept {
        // release the current object's resources
        delete[] address;
        size = 0;
        // swap from other and empty it just like in the copy constructor
        // ...
    }
}

move semantics - usage

MemoryPage factory() {
    MemoryPage p;
    // ... more code that confuses compiler to not trigger NRVO
    return p;
}
MemoryPage p = factory(); // copy construction  or
                          // move construction if we provide move ctor (which we did)
std::vector<MemoryPage> pages;
;
pages.push_back(MemoryPage{});    // avoid copy, insert directly
pages.push_back(MemoryPage{});    // again

move semantics - forcing

lvalues don’t trigger move semantics, but they can be forced to with std::move (which is a fancy static_cast to &&)

std::vector<MemoryPage> pages;
;
MemoryPage p{};
pages.push_back(p);                // calls copy constructor (even if we have move ctor) !!!
pages.push_back(std::move(p));     // calls move constructor (and p is now useless on its own)
MemoryPage p = other;              // always copy construction
MemoryPage p = std::move(other);   // move construct - Thief :)
Note
 UE4 version of std::move is called MoveTemp

move semantics - forcing (continued)

Within functions, arguments are always lvalues, even if originally at the call site a rvalue was passed

void func(MemoryPage&& input) {    // clearly it's rvalue
     MemoryPage p = input;         // copy :(
}
;
func(MemoryPage{});                // pass temporary
void func(MemoryPage&& input) {
    MemoryPage p = std::move(input);
}
Tip
 Remember: a named rvalue reference is treated as lvalue (and implicit conversion from lvalue to rvalue reference is forbidden by the standard, that’s why the need for std::move cast)

move semantics - std::move - a study in C++11

Let’s see a typical implementation of std::move

template<class T>
constexpr auto move(T&& arg) noexcept
    -> typename remove_reference<T>::type&&
{
    return static_cast<typename remove_reference<T>::type&&>(arg);
}
template<class T> struct remove_reference      { typedef T type; };
template<class T> struct remove_reference<T&>  { typedef T type; };
template<class T> struct remove_reference<T&&> { typedef T type; };
;
// The reason this is needed is because of the special rules of template argument deduction (the perfect forwarding mechanic)
// Otherwise move(lvalue_reference) wouldn't work

move semantics - recap

  • move helps with performance, optimise temporaries passed around
  • you need to have move constructors (sometimes the default may work)
  • use them when you want transfer of resources instead of copy
    • this mean you are willing to allow stealing, letting go of previous data
  • they don’t always kick in - make sure you nudge it with std::move

new libraries

Everything lives under the std:: namespace

  • #include <initializer_list>
    • initializer_list<T>
    • used to consume T = { … } etc
  • #include <utility>
    • move for forcing move semantics
    • forward for passing function arguments while preserving lvalue/rvalue’ness
    • swap specialisations for various STL types (string, vector, etc)
  • include <tuple>
    • to be discussed separately
  • #include <functional>
    • function wraps callable object of any type with specified function call signature
    • hash provides hashes for built in types and strings
    • more constructs or functors like less<>, plus<> that can be more streamlined than lambda’s sometimes

new libraries (continued)

  • #include <memory>
    • smart pointers - to be discussed separately
    • allocators - class templates encapsulating memory allocation strategy
  • #include <array>
    • array container that encapsulates fixed size arrays
    • no overhead to classic C-style array T[N] but safer: doesn’t decay to T* for ex
  • #include <unordered_???>
    • associative containers (set, map, etc) that are not sorted, allowing optimisations
    • unordered_map is a hash-table (map is not)
  • #include <chrono>
    • clock and time duration management - to be discussed separately
  • #include <random>
    • classes that generate random and pseudo-random numbers - to be discussed separately

new libraries (continued)

  • #include <ratio>
    • compile-time rational arithmetic support
    • example: ratio_equal<ratio<2,3>, ratio<4,6>>::value
  • #include <cfenv>
    • control floating-point environment: rounding, overflow, division by zero, etc
  • #include <regex>
    • dealing with regular expressions
    • basic_regex, match_results, regex_search etc
  • #include <type_traits>
    • compile-time template-based interface to query or modify the properties of types
    • is_void, is_floating_point, is_enum, is_pointer etc
    • is_lvalue_reference, is_rvalue_reference
    • is_const, is_pod, is_polymorphic, is_base_of etc
    • remove_const, remove_reference, make_signed, etc

new libraries (continued)

  • #include <thread>
    • cross-platform support for threads
    • utilities for the current thread: yield, sleep_for etc
  • #include <mutex>
    • synchronization between threads: mutex, recursive_mutex etc
    • mutex management: unique_lock, lock_guard, etc
    • call_once call a function exactly once, even if called from several threads
  • #include <condition_variable>
    • synchronization primitive that allows multiple threads to communicate with each other
    • via .wait() and .notify_one()/.notify_all()
  • #include <future>
    • async runs a function asynchronously (new thread or at a later time) and returns a future that will hold the result
  • #include <atomic>
    • atomic<> integral types wrapper that allows lockless programming
    • memory_order specifies how regular, non-atomic memory accesses are to be ordered around an atomic operation

std::tuple

A kind of generalized anonymous struct. For only 2 members there’s also std::pair.

Useful to return multiple values from functions (ex: data + error flag)

/* tuple */                                      /* struct equivalent */
std::tuple<int, float, std::string>              struct tuple {
                                                     int _0;
                                                     float _1;
                                                     std::string _2;
                                                 };
/* tuple */                                      /* struct equivalent */
auto t = std::make_tuple(1, 3.14f, "hello"s);    auto t = tuple {1, 3.14f, "hello"};
/* tuple */                                      /* struct equivalent */
std::get<0>(t);                                  t._0;
std::get<float>(t); // C++14 only                no equivalent
Tip
 UE4 equivalent is TTuple

std::tuple - examples

std::tuple<int, int, int, int> CalculateSize() {
    // logic ...
    return std::make_tuple(x, y, width, height);
}
std::tuple<Widget, bool> GetWidget() {
    Widget work{};
    // logic ...
    if (error) {
        return std::make_tuple(Widget{}, false);
    } else {
        return std::make_tuple(std::move(work), true);
    }
}
auto w = GetWidget();
if (std::get<bool>(w)) {
    std::get<Widget>(w).method();
}

std::unique_ptr

Smart pointer that retains sole ownership of an object and destroys it when it goes out of scope.

std::unique_ptr<int> ptr { new int {42} };
std::unique_ptr<int> ptr = std::make_unique<int>(42); // since C++14
std::unique_ptr<int []> ptr { new int [10] };
std::unique_ptr<int []> ptr = std::make_unique<int []>(42); // 42 elements !!!
int i = *ptr;
ptr[5] = 0;
ptr->method(); // given unique_ptr<Widget> ...
if (ptr) { ... }
if (ptr.get() != nullptr) { ... }

std::unique_ptr (continued)

No two unique_ptr instances can manage the same object.

No overhead just like normal pointer but can optionally store a custom deleter function.

Typical usage:

  • ownership (acquiring / passing) of dynamically allocated objects
  • safety by guaranteeing deletion on both
    • normal exit
    • exit through exception/error
  • as the element type in move-aware containers, such as std::vector, which hold pointers to dynamically-allocated objects
Tip
 UE4 equivalent is TUniquePtr.

std::unique_ptr - examples

{ // begin scope
    std::unique_ptr<int> ptr_1 = std::make_unique<int>(42);

    std::unique_ptr<int> ptr_2 = ptr_1; // Compile error!!! cannot copy

    std::unique_ptr<int> ptr_3 = std::move(ptr_1);
    // Transfers ownership.
    // ptr_3 now owns the memory and ptr_1 is rendered invalid.

} // end of scope: ptr_3 will be destroyed, ptr_1 is empty so nothing happens

std::unique_ptr - another example

void do_work()
{
    auto file_closer = [](FILE* f) {
        ::fclose(f);
    };

    std::unique_ptr<FILE, decltype(file_closer)> file = { ::fopen("output.bin", "wb"), file_closer };

    // more code...

    if (some_condition) {
        ::fwrite(&data, sizeof(data), 1, file.get());
    } else {
        return; // file closed!
    }

    // more code...

} // end of scope: file will be closed gracefully

std::shared_ptr

Smart pointer that retains shared ownership of an object.

The referenced object will be destroyed only when all copies of the shared_ptr have been destroyed.

std::shared_ptr<int> p1 { new int {42} };
std::shared_ptr<int> p1 = std::make_shared<int>(42);

More overhead than normal pointer (due to need of storing the reference count). Can also contain a custom deleter function.

Typical usage:

  • safe shared ownership of data
  • multi-threaded safety of shared access (but not modification)
Tip
 UE4 equivalent is TSharedPtr.

std::weak_ptr

Smart pointer that holds a non-owning ("weak") reference to an object that is managed by std::shared_ptr.

Models temporary ownership:

  • when an object needs to be accessed only if it exists
  • and it may be deleted at any time by someone else

To actually be used it will be converted into a temporary std::shared_ptr.

Tip
 UE4 equivalent is TWeakPtr.

std::shared_ptr/weak_ptr - examples

{
    std::shared_ptr<int> p1 = std::make_shared<int>(5);

    {
        std::shared_ptr<int> p2 = p1;  // add a new reference
        std::weak_ptr<int> wp1 = p1;   // doesn't count as new reference

        int n = *p2;                   // dereference like normal pointer

        int q = wp1.expired() ? 0 : *(wp1.lock()); // safe dereference

    } // end of this scope, p2 and wp1 destroyed but not the owned object

} // p1 is destroyed and so is the object

random - old way

C++03 only had srand() and rand() which have a lot of problems:

#include <stdlib.h>
#include <time.h>
srand(time(NULL));
int i = rand() % 100;
  • Lots of problems:
  • seeding based on time() has only 1Hz frequency
  • 32 bit seed only, weird unsigned to time_t warnings
  • rand() by standard only has [0..32767] range
  • non-uniform distribution, applying % can skew probabilities
  • uses Linear Congruential Generator which is weak and has low period
  • hard to convert to floats - needs bad hacks like rand() * 1.0 / RAND_MAX

random - new and better way

C++11 introduces a new paradigm by separating engines and distributions:

Random number engines - algorithms that produce integer sequences:

  • linear_congruential_engine
    • already prepared typedefs: minstd_rand, …
  • mersenne_twister_engine
    • already prepared typedefs: mt19937, mt19937_64
  • subtract_with_carry_engine
    • already prepared typedefs: ranlux24
  • random_device
    • non-deterministic true random number generator using hardware entropy source

Random number distributions - convert the output of engines:

  • uniform_int_distribution, uniform_real_distribution, etc
  • bernoulli_distribution, poisson_distribution, etc
  • normal_distribution, discrete_distribution, etc

random - example

std::random_device rd;  // can be implemented with hardware support
                        // otherwise reverts to PRNG
int r = rd();           // has limited entropy pool
                        // better used to just seed other RNG's
std::mt19937 engine(rd);
std::mt19937 engine(1779);
std::uniform_int_distribution<> dist(1, 6);  // simulate a die
int r = dist(engine);                        // get a random throw
int q = dist(engine);                        // another one
/* generate between [0..1) with 10 bits of randomness */
double chance = std::generate_canonical<double, 10>(engine);

std::chrono

Library defines three main types (durations, clocks, and time points) as well as utility functions and common typedefs.

A clock consists of a starting point (or epoch) and a tick rate. C++11 defines three clock types:

  • system_clock
    • wall clock time from the system-wide realtime clock
    • can be adjusted from C-style std::time_t
  • steady_clock
    • monotonic clock that will never be adjusted
    • good for measuring time intervals
  • high_resolution_clock
    • most accurate available with the smallest tick period
    • at the very worst it’s just an typedef for the others

time_point is an interval of time that has passed since the epoch of a specific clock. Typedef’ed per clock type.

duration consists of a count of ticks of type T and a tick period. Can be obtained from 2 time_point

std::chrono - examples

using clock_t = std::chrono::system_clock;
clock_t::time_point now = clock_t::now();

std::time_t now_c = clock_t::to_time_t(now);
std::cout << std::put_time(std::localtime(&now_c), "%F %T"); // prints <date> <time>
using clock_t = std::chrono::high_resolution_clock;

clock_t::time_point start = clock_t::now();
// expensive operation ...
clock_t::time_point end = clock_t::now();

std::chrono::duration<float, std::chrono::seconds::period> d = end - start;
std::cout << "Time taken: " << d.count();
